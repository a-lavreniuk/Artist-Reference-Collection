# Шаблон описания PR (ARC)

Проверить текст **глазами ревьюера без контекста задачи**. Улучшить, не добавляя фактов, которых не было у автора.

## Обязательные блоки

### Заголовок

Кратко: **что** + **где** (раздел или слой).

Пример: `Gallery: fix filter reset on empty search`

### Summary

- **Что изменилось** для пользователя (1–3 предложения)
- **Зачем** — какая проблема или задача
- **Подход** — если выбор неочевиден (1 предложение)

### Scope

- Слои: renderer / main / preload / storage
- Затронутые экраны или модули

### Figma (UI PR)

- Node ARC-2: `1414:13470` и т.п.
- Что сверять: отступы, состояния, типографика

### Risks & regression areas

- Общие компоненты (`EmptyState`, `ContextMenu`, gallery hooks)
- IPC каналы (если менялись)
- Migration / library path / backup

### Test plan

Пошагово, что кликнуть / проверить вручную:

```
1. Открыть библиотеку с N карточками
2. …
3. Edge: пустая коллекция, отмена импорта, …
```

### Out of scope

Что **не** входит в PR — чтобы ревьюер не ждал.

## Чеклист качества описания

- [ ] Понятно без знания чата/AnyType
- [ ] Есть ручная проверка, не только «npm test»
- [ ] Указаны риски для regression
- [ ] UI PR — Figma node или явно «без макета, по UI-Kit»
- [ ] Нет секретов, путей к локальной машине автора

## Промпт для ИИ (улучшение текста)

```
Проверь описание PR глазами ревьюера без контекста задачи.

Заголовок: …
Описание: …

Ответь:
1. Чего не хватает для понимания что/зачем?
2. Какие риски и ручные проверки стоит добавить?
3. Предложи улучшенный текст, сохранив смысл автора — без выдуманных деталей.
```

## Связка с arc-code-review

После Bugbot/Security — в PR description включить «Known limitations» если critical не блокирует merge по решению команды.
